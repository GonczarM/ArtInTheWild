# Migration Notes — Phase 0: Inventory & Safety Net

Generated by reading `server.js`, `controllers/`, `models/`, `config/`, `.env`, and the
frontend API utils in `src/utils/`. No code was changed to produce this document.

**Note on naming:** the migration brief refers to the domain entity as "Art piece," but
in this codebase it's called **Mural** throughout (`Mural` model, `/api/murals` routes,
`muralSearch`, etc.). This document uses the actual names — Strapi content type planning
in Phase 2 should use "Mural" to match, or explicitly decide to rename.

---

## 1. Express Routes

Mounted in `server.js`:
- `app.use('/api/users', usersController)`
- `app.use('/api/murals', muralsController)`
- Global `checkToken` middleware (`config/checkToken.js`) runs on **every** request: it
  reads `Authorization` header or `?token` query param, verifies the JWT, and sets
  `req.user` to the decoded payload (or `null` if missing/invalid). It never blocks the
  request itself — auth enforcement is opt-in per route via `ensureLoggedIn`.
- Catch-all `app.get('/*', ...)` serves `dist/index.html` (Vite SPA fallback).
- Centralized error handler maps `error.status === 403` → 403 "Forbidden",
  `error.code === 11000 || error.status === 409` → 409 "Conflict" (used for Mongo
  duplicate-key errors, e.g. duplicate username), else → 400 with the raw error object.

### `/api/murals` (controllers/murals.js)

| Method | Path | Auth | Notes / Request / Response |
|---|---|---|---|
| POST | `/` | Optional (sets `user` owner if logged in) | `multipart/form-data` via `upload.single('photo')` (S3). Body becomes the Mural fields; if a photo file is present, `photos: [{photo: <s3 url>}]` and `favoritePhoto` are set. Resp: `{ mural }` |
| GET | `/seed` | None | One-off/dev route. Fetches Chicago public mural dataset (`https://data.cityofchicago.org/resource/we8h-apcf.json`), dedupes by title, bulk-creates `Mural` docs. Resp: `{ mural: createdMurals }`. **Likely not part of the real product contract — flag for Strapi (could become a seed script instead of a live route).** |
| GET | `/` | None | All murals. Resp: `{ murals }` |
| GET | `/search/:type/:term` | None | Regex (case-insensitive) search: `Mural.find({ [type]: { $regex: term, $options: 'i' } })`. `type` is any field name (e.g. `title`, `artist`, `zipcode`) — passed straight through from the URL with no allowlist. Resp: `{ murals }` |
| GET | `/list/:type/:term` | None | Same regex query as above, but returns up to 10 **distinct values** of `type` field (autocomplete/suggestion list) instead of full docs. Resp: `{ searchList }` |
| GET | `/photo` | None | Murals where `photos` array exists and is non-empty (used for map view). Resp: `{ murals }` |
| GET | `/:id` | None | Single mural by id. Resp: `{ mural }` |
| PUT | `/:id` | **Required** (`ensureLoggedIn`) + owner check (`req.user._id === mural.user`, else 403) | Full-body update via `findByIdAndUpdate`. Resp: `{ mural }` |
| PUT | `/photo/:id` | **Required** + `upload.single('photo')` | Appends `{photo: <s3 url>}` to `photos[]`; sets `favoritePhoto` if unset. **No owner check** — any logged-in user can add a photo to any mural. Resp: `{ mural }` |
| DELETE | `/:id` | **Required** + owner check (else 403) | Resp: `{ status: 200, mural: deletedMural }` |

### `/api/users` (controllers/users.js)

| Method | Path | Auth | Notes / Request / Response |
|---|---|---|---|
| POST | `/register` | None | Body: `{ username, password }`. Creates `User`, returns a signed JWT directly as the response body (not wrapped in JSON object — `res.json(token)`). |
| POST | `/login` | None | Body: `{ username, password }`. bcrypt-compares password, returns JWT (same raw-string shape as register). **Bug to note for parity decisions:** when the user isn't found or password doesn't match, `next(error)` is called but execution falls through (no `return`) — it still attempts `bcrypt.compare` / `createJWT` on a possibly-null `foundUser`, which would throw and produce a second `next()` call. Strapi's built-in auth won't have this issue, so likely moot, but flagging since it's a real bug in current prod behavior. |
| GET | `/murals` | Implicit (`req.user._id` used directly, no `ensureLoggedIn` guard) | Murals owned by current user. **Will throw if not logged in** (`req.user` is `null`) — effectively requires auth but fails with a 400/500 rather than a clean 401. |
| GET | `/favorites` | Implicit (same issue as above) | Murals where `favorite` array contains current user's id. |
| PUT | `/favorite/:id` | **Required** (`ensureLoggedIn`) | Pushes `req.user._id` into mural's `favorite[]` array (no de-dupe check — can favorite multiple times). Resp: `{ mural }` |
| PUT | `/photo/:id` | **Required** | `:id` here is a **photo subdocument id**, not a mural id — finds the mural containing that photo, pushes a like, recomputes `favoritePhoto` as whichever photo has the most likes. Resp: `{ mural }` |
| DELETE | `/` | **Required** | Deletes current user and cascades — deletes all murals owned by that user (`Mural.deleteMany({user: req.user._id})`). **Destructive cascade — no orphaning of murals to "deleted user," they're hard-deleted.** Decide deliberately whether Strapi should replicate this or orphan/reassign instead. |

### Auth model summary
- Stateless JWT, no server-side session store. Token payload is `{ user: <full user doc minus password> }`, 24h expiry, signed with `SECRET`.
- Client stores token in `localStorage` (`src/utils/users-service.js`) and sends as `Authorization: Bearer <token>` on every request (`src/utils/send-request.js`).
- This maps reasonably well to Strapi's `users-permissions` plugin (Phase 2), which also issues bearer JWTs — but the "decoded payload is the user doc" pattern won't carry over (Strapi just gives an id, frontend calls `/api/users/me`).

## 2. Mongoose Schemas

### `User` (models/user.js)
| Field | Type | Constraints |
|---|---|---|
| `username` | String | required, unique, trim |
| `password` | String | required, trim, minLength 3; bcrypt-hashed in a `pre('save')` hook (6 salt rounds); stripped from `toJSON` output |
| `createdAt`/`updatedAt` | timestamps | auto |

No `email`, `role`, or profile fields — Strapi's `users-permissions` User model has more
fields than this (email, confirmed, blocked, role relation) — decide whether to
backfill/ignore the extras.

### `Mural` (models/mural.js)
| Field | Type | Constraints |
|---|---|---|
| `title` | String | required |
| `artist` | String | optional |
| `description` | String | required |
| `year` | Number | optional |
| `affiliation` | String | optional |
| `address` | String | required |
| `zipcode` | String | required |
| `latitude` | Number | required |
| `longitude` | Number | required |
| `favorite` | [ObjectId ref User] | array of user ids who favorited this mural (acts as the **Favorites join**, embedded rather than a separate collection) |
| `user` | ObjectId ref User | owner/creator; **not required** at schema level (anonymous mural creation is allowed per the controller) |
| `photos` | [photosSchema] (embedded subdocs) | see below |
| `favoritePhoto` | String | S3 URL of the currently-most-liked (or first) photo |
| `createdAt`/`updatedAt` | timestamps | auto |

### `photosSchema` (embedded in Mural, models/mural.js)
| Field | Type | Constraints |
|---|---|---|
| `photo` | String | S3 URL |
| `likes` | [ObjectId ref User] | per-photo like tracking |

### Relationships
- **User → Mural**: one-to-many via `Mural.user` (optional/nullable owner).
- **User ↔ Mural favorites**: many-to-many, but modeled as an array on `Mural.favorite`
  (not a separate join collection, and not present at all on the `User` schema — you
  can't query "this user's favorites" from the User side, only via
  `Mural.find({favorite: userId})`). Strapi will likely want a proper many-to-many
  relation field on both sides, or a join content type.
- **Mural → Photos**: one-to-many, embedded subdocuments (not a separate collection).
  Each photo has its own `likes` array (User many-to-many) and its own implicit `_id`
  used directly as a route param (`PUT /api/users/photo/:id`). Strapi's media library
  treats uploads as a flat collection with relations, not nested-with-likes subdocuments
  — **this is the field that won't map cleanly; flag explicitly in Phase 2** (likes-per-photo
  needs its own content type/relation, e.g. a `Photo` content type owned by `Mural` with
  a `likes` many-to-many to `User`, since Strapi has no embedded-array-with-relations
  equivalent).
- `favoritePhoto` is denormalized (a copy of a URL already in `photos[]`), recomputed
  server-side whenever a like is added. Worth deciding whether to keep as denormalized
  data or compute on read in Strapi.

## 4. Open Questions — Decisions (2026-06-30)

1. **Photo likes** (`photos[].likes`) — **Tabled.** No Strapi content-type redesign yet;
   revisit when we actually build the Photo content type in Phase 2.
2. **`GET /api/murals/seed`** — **Confirmed dev-only.** Will not be carried over as a live
   Strapi route. Convert to a one-off seed/migration script (Phase 3) instead.
3. **`S3_BASE_URL`** — **Confirmed unused, drop it.** Strapi's S3 upload provider
   generates its own URLs; don't carry this var into Strapi's env config.
4. **Cascade delete** (`DELETE /api/users` removes all of that user's murals) —
   **Changing to orphaning.** Deleting a user should leave their murals in place with
   `user` unset, not hard-delete them. This is a behavior change from current Express
   behavior, not a 1:1 port — implement orphaning in the Strapi lifecycle/permissions
   layer in Phase 2, not in Express (old backend stays untouched per the brief).
5. **Anonymous mural creation** (`Mural.user` currently optional) — **Requiring login
   going forward.** New Strapi permissions should require auth to create a mural. Any
   *existing* murals with no `user` set need to be flagged during Phase 3 data migration
   so a placeholder/legacy owner can be assigned (or left orphaned consistently with
   decision #4 — same shape of data either way, decide assignment at migration time).
6. **`PUT /api/murals/photo/:id` has no owner check** — **Confirmed intentional, keep
   as-is.** Any authenticated user may add a photo to any mural (collaborative photo
   contributions). Strapi permissions for the photo-add action should allow any
   authenticated user, not just the mural owner.
7. **Search routes accept arbitrary field names** (`/search/:type/:term`,
   `/list/:type/:term`) — **Confirmed only `title`, `artist`, and `zipcode` are actually
   used by the frontend.** Drop the open-ended arbitrary-field version; Strapi
   filters/custom routes only need to cover these three fields.
